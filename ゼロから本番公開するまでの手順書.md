## 1. このアプリが何をするか

ヤフオクの検索条件を登録しておくと、新着商品が出た瞬間にスマホへプッシュ通知を送る監視ツール。

**特徴:**
- Swift/Kotlin不使用。Next.jsだけで作ったPWA（Progressive Web App）
- ブラウザだけでホーム画面にインストールでき、見た目はネイティブアプリと同じ
- サーバー費用ゼロ（〜50人まで無料枠だけで運用できる）
- GitHub Actionsが10分ごとにヤフオクを自動スクレイピングして通知を送る

## 2. 全体のセットアップ手順

**所要時間: 慣れれば1〜2時間**

大きな流れはこの6ステップ：
1. GitHubにコードを置く（cronの実行場所になる）
2. Supabaseでデータベースを作る（ユーザーデータや通知履歴を保存する場所）
3. VAPIDキーを生成する（プッシュ通知を送るための証明書みたいなもの）
4. Vercelでアプリを公開する（誰でもURLからアクセスできるようにする）
5. 各サービスに秘密の鍵を登録する（1〜4がお互いに話せるようにする）
6. 動作確認する（実機で通知が届くか確かめる）

---

### ① GitHub（コードの置き場所 + cronの実行場所）

**アカウント作成**
1. https://github.com を開いてSign upする
2. メールアドレスとパスワードを設定して登録完了

**リポジトリ（コードの入れ物）を作る**
1. 右上の「+」→「New repository」をクリック
2. Repository nameに好きな名前を入れる（例: yahooauction-watch）
3. **Publicを必ず選ぶ**（Privateにすると月2,000分しかcronが動かない。10分ごとのcronは月4,320回走るので無料枠を大幅超過する。Publicなら無制限）
4. 「Create repository」をクリック

**コードをGitHubに送る**

ターミナルでプロジェクトフォルダに移動し、以下の順番で操作する。まずgitの初期化（git init）、次に全ファイルを追加してコミット、ブランチ名をmainに設定、GitHubのリポジトリURLをremoteに追加して、最後にpushする。

---

### ② Supabase（データベース）

**アカウント作成**
1. https://supabase.com を開いてSign upする
2. 「Continue with GitHub」でGitHubアカウントと連携するのが一番ラク

**プロジェクト（データベース）を作る**
1. 「New project」をクリック
2. Project nameに好きな名前を入れる（例: yahooauction-watch）
3. Database passwordに強いパスワードを設定する（**どこかにメモしておく**）
4. Regionは「Northeast Asia (Tokyo)」を選ぶ（日本に近いので速い）
5. 「Create new project」をクリック → 1〜2分待つ

**テーブルを一括作成する（これ1回で全部終わる）**
1. 左メニューの「SQL Editor」を開く
2. このプロジェクトの `supabase/setup_all.sql` の中身を全部コピーして貼り付ける
3. 右下の「Run」ボタンを押す
4. 「Success. No rows returned.」が出れば完了

**必要な情報をメモする**
左メニューの「Settings」→「API」を開くと以下が見つかる：

| 名前 | どこにあるか | 後で使う変数名 |
|------|-------------|--------------|
| Project URL | 「Project URL」の欄 | NEXT_PUBLIC_SUPABASE_URL |
| anon public キー | 「Project API keys」のanon | NEXT_PUBLIC_SUPABASE_ANON_KEY |
| service_role キー | 「Project API keys」のservice_role | SUPABASE_SERVICE_KEY |

> ⚠️ service_roleキーは絶対に外部に漏らさない。GitHubにコミットしない。

---

### ③ VAPIDキー生成（プッシュ通知用の証明書）

ターミナルで `npx web-push generate-vapid-keys` を1回だけ実行する。

実行すると「Public Key」と「Private Key」という2つの長い文字列が表示される。この2つをどこかにメモしておく。

**この2つは後で絶対に変えない。変えると全ユーザーの通知が届かなくなる。**

| 名前 | 変数名 |
|------|--------|
| Public Key | VAPID_PUBLIC_KEY |
| Private Key | VAPID_PRIVATE_KEY |

---

### ④ Vercel（アプリの公開）

**アカウント作成**
1. https://vercel.com を開いてSign upする
2. 「Continue with GitHub」でGitHubアカウントと連携する

**プロジェクトを作成してデプロイする**

方法A: Vercelダッシュボードから（GUI操作）
1. ダッシュボードの「New Project」をクリック
2. GitHubのリポジトリ一覧から対象のリポジトリを選んで「Import」
3. Framework Presetが「Next.js」になっているか確認
4. 「Environment Variables」セクションで⑤の表の変数を全部追加する（後で追加もできる）
5. 「Deploy」をクリック → 1〜2分で完了

方法B: ターミナルから（CLI操作）

`npm install -g vercel` でVercel CLIをインストール（初回のみ）し、`vercel login` でブラウザ経由でログインする。その後、プロジェクトフォルダで `vercel --prod` を実行するとデプロイされる。

**PWAに必要なvercel.jsonの設定（これがないとSWが古いままになる）**

プロジェクトの vercel.json に headers 設定を追加する。sw.js と manifest.json の2ファイルに対して Cache-Control を `no-cache, no-store, must-revalidate` に設定する。sw.jsとmanifest.jsonはCDNにキャッシュさせてはいけない。この設定がないとデプロイしてもユーザーに古いSWが残り続ける。

---

### ⑤ 各サービスに秘密の鍵を登録する

ここが一番大事なステップ。VercelとGitHub Actionsの**両方**に同じ変数を登録する必要がある。

**Vercelへの登録方法**
Vercelダッシュボード →「プロジェクト名」→「Settings」→「Environment Variables」

| 変数名 | 値 | どこから取得するか |
|--------|----|--------------------|
| NEXT_PUBLIC_SUPABASE_URL | https://xxx.supabase.co | Supabase → Settings → API |
| NEXT_PUBLIC_SUPABASE_ANON_KEY | eyJ... | Supabase → Settings → API |
| SUPABASE_SERVICE_KEY | eyJ... | Supabase → Settings → API（service_role） |
| VAPID_PUBLIC_KEY | BH... | ③で生成したPublic Key |
| VAPID_PRIVATE_KEY | xo... | ③で生成したPrivate Key |
| NEXT_PUBLIC_APP_URL | https://xxx.vercel.app | Vercelのデプロイ後のURL |
| CRON_SECRET | 自分で決めた任意の文字列 | 例: my-secret-2026 |

追加したら「Save」→ Vercelのダッシュボードから「Redeploy」

**GitHub Actionsへの登録方法**
GitHub のリポジトリページ →「Settings」→「Secrets and variables」→「Actions」→「New repository secret」

| シークレット名 | 値 |
|---------------|----|
| SUPABASE_URL | Supabase の Project URL |
| SUPABASE_ANON_KEY | Supabase の anon キー |
| SUPABASE_SERVICE_KEY | Supabase の service_role キー |
| VAPID_PUBLIC_KEY | ③で生成したPublic Key |
| VAPID_PRIVATE_KEY | ③で生成したPrivate Key |

> ⚠️ VercelとGitHub Actionsで**VAPIDキーは必ず同じ値**を使うこと。
> 違う値を入れると、GitHubActionsからの通知がユーザーに届かなくなる。

1. Vercelのデプロイが完了したら、公開URLにアクセスする
2. iPhoneのSafariで開いて「ホーム画面に追加」→ アプリとして起動

## 3. サービス別の詳細と選んだ理由

### Vercel

| 項目 | 内容 |
|------|------|
| 用途 | Next.jsアプリのホスティング + API Routes（サーバーレス関数） |
| 無料枠 | Hobbyプラン: 帯域幅100GB/月、ビルド6,000分/月 |
| 有料プラン | Pro $20/月: 帯域幅1TB、チームコラボ |
| なぜ選んだ | Next.jsと最も親和性が高い。デプロイが1コマンド。HTTPS自動対応 |
| 重要な仕組み | VERCEL_GIT_COMMIT_SHA 環境変数がデプロイごとに自動で変わる → SWがこれを見て自動キャッシュ更新 |

---

### Supabase

| 項目 | 内容 |
|------|------|
| 用途 | PostgreSQL DB（ログイン機能は使わず、ブラウザのUUIDで独自管理） |
| 無料枠 | DB 500MB、帯域幅2GB/月、MAU 50,000人。2週間アクセスなしで一時停止（cronが動いていれば停止しない） |
| 有料プラン | Pro $25/月: DB 8GB、帯域幅50GB、一時停止なし |
| なぜ選んだ | 無料枠が充実。REST API自動生成。管理UIが使いやすい |

**テーブル設計（4テーブル構成）:**

- ユーザーテーブル（users）: ユーザーID + 通知設定（ntfy / Discord / WebPush）
- 検索条件テーブル（conditions）: キーワード・価格・フィルター等
- 通知済みIDテーブル（notified_items）: 重複通知防止用。7日で自動クリーンアップ
- 通知履歴テーブル（notification_history）: 履歴ページ表示用。72時間で自動クリーンアップ

---

### GitHub Actions（cron）

| 項目 | 内容 |
|------|------|
| 用途 | 10分ごとにヤフオクをスクレイピングして通知を送るバッチ処理 |
| 無料枠 | **パブリックリポジトリ: 無制限** / プライベート: 2,000分/月 |
| 実行内容 | scriptsフォルダのrun-check.tsをNode.jsで実行する |

cronの設定は .github/workflows のYAMLファイルで行う。スケジュールを「10分ごと」に設定し、手動実行ボタンも追加している。

> ⚠️ プライベートリポジトリにした場合、月約8,000〜13,000分必要でGitHub Pro（$4/月・3,000分）でも全然足りない。その場合はVercel Cron（Pro $20/月）かVPSに移行が必要。

---

### Web Push / VAPID（プッシュ通知インフラ）

| 項目 | 内容 |
|------|------|
| 用途 | スマホのロック画面にリアルタイム通知を送る |
| コスト | **完全無料** — Apple APNs・Google FCMのインフラを経由するため費用ゼロ |
| ライブラリ | web-push npm パッケージ |
| 仕組み | VAPID鍵ペアで認証 → ブラウザのPush Serviceに送信 → Apple/Googleが端末に配信 |
| iOS制限 | iOS 16.4以上 + Safari + ホーム画面追加（PWAモード）が必須 |

**iOS向けに必須のヘッダー設定（これがないと即時配信されない）:**

通知送信時に urgency を high に設定し、APNs専用のヘッダーとして `apns-push-type: alert`（backgroundにするとロック画面に表示されない）と `apns-priority: 10`（最高優先度。これがないと通知が遅延する）を必ず付ける。

---

### ntfy.sh（代替通知チャンネル）

| 項目 | 内容 |
|------|------|
| 用途 | Web Pushが使えないユーザー向けの代替通知。ntfyアプリをインストールするだけで使える |
| コスト | 無料（パブリックトピックはレート制限あり: 250回/時） |
| 注意点 | タイトルに日本語を使うとHTTPヘッダーでUnicodeエラーが出る |

**日本語タイトルをHTTPヘッダーに入れない実装:**

日本語のタイトルはHTTPヘッダーに直接入れると文字化けエラーになる。代わりにURLのクエリパラメータとして渡す。こうすることでエラーなく日本語通知が送れる。

---

### Discord Webhooks（代替通知チャンネル）

| 項目 | 内容 |
|------|------|
| 用途 | DiscordサーバーのチャンネルにリッチなEmbedで通知 |
| コスト | 無料・無制限 |
| セキュリティ | SSRF対策として discord.com / discordapp.com のホスト名のみ許可するコードを実装済み |

## 5. データの流れ（詳細）

### GitHub Actionsのバッチ処理（メインフロー）

1. Supabase から enabled=true の全条件を取得する
2. ユーザー情報を一括取得して通知設定を確認する
3. 同じキーワード＋価格の条件をグループ化する（同じ検索は1回だけフェッチして効率化）
4. Yahoo Auctions のHTMLをフェッチする（iPhoneのSafariに偽装したUser-Agentを使い、モバイル版HTMLを取得する）
5. data-auction-で始まる属性をパースして商品情報を組み立てる
   - data-auction-id: オークションID
   - data-auction-price: 現在価格
   - data-auction-title: 商品名
   - data-auction-img: サムネイル
   - data-auction-buynowprice: 即決価格（0はなし、正の数が即決価格）
   - data-auction-endtime: Unix形式のタイムスタンプ → 残り時間を計算
6. Supabase の notified_items テーブルで重複チェックをする
7. 未通知のアイテムに通知を送信する（Web Push → Apple APNs / Google FCM → 端末 / ntfy.sh → ntfyアプリ / Discord → Discordチャンネル）
8. notified_items と notification_history に記録する
9. クリーンアップを実行する
   - 終了オークションを通知履歴から削除（1回の実行で最大10件チェック）
   - 72時間を超えた通知履歴を削除
   - 7日を超えた通知済みレコードを削除

### ユーザーの操作フロー

初回アクセス時:
- ブラウザの localStorage にUUIDを生成する（これがユーザーID）
- Supabase に users レコードを作成する
- オンボーディング: ヤフオクを Safari で開いてログインを誘導する
- Web Push の許可リクエストを表示する（iOSはホーム画面追加後のみ許可できる）
- push_sub（端末の受信先情報）を Supabase に保存する

条件追加時:
- 条件追加APIを呼び出してSupabase の conditions テーブルに挿入する

手動チェック実行時（ウォッチ画面からボタンで）:
- 即時スクレイピング＋通知送信を実行する（最大5件/条件）
- レート制限として1分に3回まで実行可能（スパム防止）

### Service Workerのキャッシュ更新フロー（2層構造）

デプロイしても古いバージョンが残り続ける問題を解決した仕組み:

第1層（手動管理）: sw.js の先頭に CACHE_VERSION という変数がある。大規模改修時だけ手動で番号を上げる。この値が変わると sw.js ファイル自体が変わり、全ユーザーのService Workerが強制更新される。

第2層（自動管理）: /api/version というエンドポイントがある。Vercelがデプロイするたびに変わるコミットIDを返す。Service Worker は起動時にこのエンドポイントを確認し、前回保存したIDと違えば全キャッシュを削除して全タブに更新シグナルを送信する。アプリ側がそのシグナルを受け取り、画面を自動リロードする。

デプロイ後にユーザーがアプリを開くと、この仕組みが自動的に動いて最新版に切り替わる。

## 6. なぜSwiftやAndroidアプリを作らなかったのか

### PWAで「ほぼ同等」が実現できるから

| 比較軸 | ネイティブアプリ | PWA（今回の選択）|
|--------|----------------|-----------------|
| 開発言語 | Swift + Kotlin（2言語必要）| TypeScript/Next.js 1本 |
| App Store 審査 | 初回 1〜4週間、毎リリースごとに審査 | なし（即時デプロイ） |
| Apple 手数料 | 有料課金の30%をAppleに取られる | なし（Stripe直接決済可）|
| 開発者登録費 | $99/年（Apple）+ 無料（Google）| なし |
| 更新の届け方 | ユーザーがApp Storeで手動アップデート | 次回起動時に自動適用 |
| バックグラウンド処理 | ネイティブAPIで自由 | 不可（GitHub Actionsで代替） |
| プッシュ通知 | フル対応 | iOS 16.4+ + Safari + ホーム画面追加が条件 |
| ホーム画面アイコン | ネイティブ | Safariの「ホーム画面に追加」で同等 |

**特にこのようなスクレイピング・自動化ツールはApp Storeの審査を通らない可能性が高い。**
Appleのガイドラインではスクレイピング系・自動化系ツールは却下されることがある。
PWAならAppleの許可が不要。

### iOSのバックグラウンド処理をどう代替したか

iOS PWAはバックグラウンドでコードを実行できない。これをGitHub Actionsに外出しすることで解決した。

処理の流れはこうなっている: GitHub Actionsが実際のスクレイピングと判定ロジックを動かし、web-pushパッケージを通じてVercelのサーバーサイドから通知を送る。Apple APNsがその通知をiPhoneに届ける。PWAはその通知を受け取るだけ。

## 7. 難しかった技術的課題

### 課題1（最難関）: Yahooの価格が取得できない

**何が起きたか:** 通知履歴の価格が全件「—」になる。3回の実装で失敗し続けた。

| 試み | 方法 | なぜ失敗したか |
|------|------|---------------|
| 1回目 | priceというCSSクラスを持つspan要素をパース | そのクラスはモバイル版HTMLに存在しない |
| 2回目 | ページ内のJSONデータを抽出 | Yahooはその構造ではなかった |
| 3回目 | Yahoo RSS フィードをパース | YahooがオークションRSSを廃止済み |

**正解:** `data-auction-price` というHTML属性に全部入っていた。

1つのオークションIDに対して2つのHTML要素が存在し、属性が分散している（一方にタイトルと画像、もう一方に価格と終了時刻）。同じIDの要素を辞書（Map）でまとめることで完全な商品情報を組み立てる。

**発見方法:** curlで実際のYahooのHTMLを取得して data-auction という文字列を検索したら一発で見つかった。

**教訓:** 推測でパーサーを書かない。まずcurlで実際のHTMLを目で見る。

---

### 課題2: iOSのオンボーディングが即座に次の画面に進む

**何が起きたか:** 「ヤフオクを開く」ボタンを押した瞬間に、Safariがまだ開いていないのに次のステップに進んでしまう。

**原因:** window の focus イベントは、ボタン要素にフォーカスが当たった時にも発火する。つまりボタンをタップするだけでfocusイベントが起き、即座に次のステップに進んでしまっていた。

**修正方法:** 4ステップで解決した。

1. focusイベントへの依存をやめる
2. ボタンを押した時刻を記録しておく
3. アプリに画面が戻ってきたタイミング（visibilitychangeイベント）で「3秒以上経過しているか」を確認する。3秒以上ならSafariから本当に戻ってきたと判定して次へ進む
4. 念のため「ログインした → 次へ進む」手動ボタンも追加する

### 課題3: Service Workerのキャッシュが古いまま残り続ける

**何が起きたか:** デプロイしても一部ユーザーに古いバージョンが表示され続ける。

**解決策:** 前述の2層構造（CACHE_VERSION手動インクリメント + /api/versionの自動検知）を組み合わせる。加えて vercel.json で sw.js 自体を no-store にしてCDNにキャッシュさせない。この設定がないと sw.js のファイル自体が古いままになる。

### 課題5: Web PushのiOS制限

iOSでのWeb Pushには厳しい条件がある：

| 条件 | 内容 |
|------|------|
| iOSバージョン | 16.4以上 |
| ブラウザ | Safariのみ（ChromeもFirefoxも裏はWebKitだが Web Push未対応） |
| インストール | 必ずホーム画面に追加が必要（ブラウザで開いているだけでは受信不可） |
| APNsヘッダー | apns-push-type: alert と apns-priority: 10 が必須 |
| VAPIDキー | 一度設定したら変更不可（変えると全購読が無効化） |

これがオンボーディングで「Safariで開いてホーム画面に追加する」フローを必須にしている理由。

## 11. 付録：ヤフオクのHTML構造（2026年4月現在）

ここを知らないと価格が取れない。実際のHTMLの仕組みを説明する。

ヤフオクの検索結果では、1つの商品のデータが2つのHTML要素に分散して入っている。

- 要素1: オークションID・商品名・サムネイル画像URLを持つ
- 要素2: 同じオークションID・現在価格・即決価格・終了時刻（Unixタイムスタンプ）を持つ

**パース方針:** 同一IDの要素を辞書（Map）でマージして完全な商品情報を組み立てる。

1ページあたり最大58件程度取得できる。価格は data-auction-price という属性に数値で入っている。即決価格は data-auction-buynowprice が0より大きい場合のみ有効で、0の場合は「即決なし」を意味する。終了時刻は data-auction-endtime にUnixタイムスタンプで入っており、現在時刻との差分で残り時間を計算する。

### 入札数の取得（重要・落とし穴あり）

`data-auction-*` 属性には入札数が含まれていない。HTMLの別の場所（`<span class="Item__bid">`）に入っており、オークションIDの位置から約5,000〜6,000文字後方にある。

```html
<span class="Item__bid">
  <span class="Item__label">入札</span>
  <span>3</span>   ← 入札あり（件数）
</span>

<span class="Item__bid">
  <span class="Item__label">入札</span>
  -               ← 入札なし（"-"で表示される。"0件"ではない）
</span>
```

**実装上の注意:**
- スキャン窓を少なくとも `pos + 10000` にしないと入札テキストに届かない（3000では届かない）
- 「入札 -」パターンを明示的に検出してbids=0と記録する必要がある
- 即決商品（isBuyItNow=true）は入札という概念がないため入札数フィルターをスキップする

### 改修1: 通知タップ・履歴タップでSafariが開かない問題

**症状:** 通知をタップしたとき・履歴の商品をタップしたとき、ヤフオクのページが「アプリ内WebView（WKWebView）」で開いてしまい、Safariにログインしているアカウントが使えない。二重ログイン状態になって入札できない。

**根本原因:** iOS PWAはスタンドアロンモードでWKWebViewを使うため、SafariのCookieとは完全に分離されている。WKWebView内でヤフオクにログインしようとすると、すでにSafariでログイン済みのアカウントとは別セッションになる。

**修正方針:**
- `window.open('https://page.auctions.yahoo.co.jp/...', '_blank', 'noopener')` でスコープ外URLを開く → iOSはSafariを起動する
- Service Workerの `notificationclick` ハンドラでも `self.clients.openWindow()` にYahooのURLを直接渡す（`/redirect/` 経由はやめる）

```javascript
// sw.js の notificationclick
const targetUrl = auctionId
  ? `https://page.auctions.yahoo.co.jp/jp/auction/${auctionId}`
  : base + '/'
event.waitUntil(self.clients.openWindow(targetUrl))
```

```typescript
// history/page.tsx の openAuction
function openAuction(_url: string, auctionId: string) {
  const yahooUrl = `https://page.auctions.yahoo.co.jp/jp/auction/${auctionId}`
  window.open(yahooUrl, '_blank', 'noopener')
}
```

**事前ログイン誘導フロー（オンボーディング）:**
初回起動時に `window.open('https://auctions.yahoo.co.jp/', '_blank', 'noopener')` でSafariを開いてログインを促す。以降の商品ページ遷移はSafariのログイン済みセッションがそのまま使われる。

### 改修2: iOS「ページを開けません」エラー（yahuoku://スキーム廃止）

**症状:** 通知タップ後に「ページを開けません」というエラーダイアログが出る。

**原因:** `yahuoku://` というカスタムURLスキームを `window.location.href` や302リダイレクトで呼び出していた。iOSのWKWebViewはカスタムスキームへのリダイレクトを拒否する。

**修正:** `yahuoku://` を完全廃止。ブラウザURL（https://page.auctions.yahoo.co.jp/...）への直接遷移に統一。iOSはhttpsのYahoo URLをSafariで開き、Safariが「ヤフオクアプリで開く」ダイアログを出すかブラウザで表示する。

### 改修3: 入札数が全件 null になり通知が来ない

**症状:** 入札件数フィルターを「1件以上」に設定しても、60件取得しているのに通知が0件になる。

**調査手順:**
1. `curl` でYahooのHTMLを実際に取得してファイルに保存する
2. `grep -n "入札"` で入札テキストの位置を確認する
3. `data-auction-id` の出現位置と `入札` テキストの出現位置の差を計算する
4. → **差分: 約5,781文字**。当時のスキャン窓は `pos + 3000` のため入札テキストに届いていなかった

**修正: lib/scraper.ts の二次パス**

```typescript
// 変更前
let scanEnd = Math.min(html.length, pos + 3000)

// 変更後
let scanEnd = Math.min(html.length, pos + 10000)  // 5781文字の距離をカバー
```

また「入札 -」パターン（0件表示）も追加した:

```typescript
if (bm) {
  attrs['bids'] = bm[1]
} else if (/入札\s*[-－]/.test(ctxText)) {
  attrs['bids'] = '0'  // 0件オークション（"-"はYahooの0件表記）
}
```

**入札数フィルターのロジック（改修後）:**

```typescript
// 即決アイテムは入札数フィルターをスキップ（入札という概念がない）
if (item.isBuyItNow === true) return true
if (minBids <= 0 && maxBids === null) return true
if (item.bids === null) return minBids <= 0      // 未検出=0件扱い
if (minBids > 0 && item.bids < minBids) return false
if (maxBids !== null && item.bids >= maxBids) return false
return true
```

**教訓:** curlで実際のHTMLを目で見て、テキストの位置を数値で確認してから実装する。

### 改修4: 入札件数の範囲フィルター追加（minBids〜maxBids）

「1件以上10件未満」のような入札数範囲指定を追加。競争が少ない穴場商品を狙うユースケース。

`max_bids` カラムが存在しない環境（migration未実行）でもINSERTが壊れないよう、nullの場合はINSERTオブジェクトから除外する:

```typescript
const insertRow: Record<string, unknown> = { /* 既存カラム */ }
if (input.maxBids !== null && input.maxBids !== undefined) {
  insertRow.max_bids = input.maxBids  // nullの場合はキー自体を省略
}
```
